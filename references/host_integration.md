# Host integration / 宿主 agent 接入片段

如果宿主环境没有原生 skill 路由，把下面片段复制到宿主 agent 的 `AGENTS.md`、system prompt 或工具路由配置中。将 `$SKILL_DIR` 替换为 Info-Alchemist skill 根目录。

```md
## Info-Alchemist routing

把 Info-Alchemist 作为联网搜索总闸门：只要准备联网搜索、查公开资料、查新闻/趋势/动态、查竞品/市场/SERP、判断方向或整理一堆信息，默认直接调用 `$SKILL_DIR/scripts/formal_run.py`。只有调试或规则不确定时再读取 `$SKILL_DIR/SKILL.md`，不要绕过它直接普通搜索。

如果用户本轮明确说“查询 / 查一下 / 搜 / 调研 / 看看有没有机会”，必须把**用户本轮原始问题**传给 `$SKILL_DIR/scripts/formal_run.py --query`。不要把它当成上一轮报告后的普通追问，不要直接要求用户在旧建议里二选一，也不要把本轮问题改写成上一轮主题或旧关键词。

以下场景尤其必须先进入 Info-Alchemist：

- “帮我查一下 ...”且背后有决策
- “最新 ... 新闻 / 趋势 / 动态”这类开放集合查询，先触发行动目的确认门；用户确认行动目的前不搜索
- “最近/热门/比较火/头部 + 工具站/网站/站点/产品/工具”这类开放集合查询，先确认行动目的；热门不是决策意图
- “... 有没有机会”
- “... 值不值得做 / 要不要做 / 能不能做”
- “有没有 SEO 机会 / 竞品信号 / 市场信号”
- “很多信息，不知道哪些重要 / 哪些该做”
- “最近做了很多，不知道有没有结果 / 要不要继续 / 要不要转向”

执行规则：
1. 默认只调用一次正式入口：`python3 $SKILL_DIR/scripts/formal_run.py --query "<用户原始问题>"`。
2. 如果上一轮是行动目的确认，本轮是选项、短确认或补充：`python3 $SKILL_DIR/scripts/formal_run.py --previous-query "<上一轮原始问题>" --reply "<用户本轮回复>" --confirmation-round <上一轮返回的 confirmation_round>`。
3. `formal_run.py` 会完成激活、行动目的确认或续跑解析、搜索计划、联网搜索、证据整理和 run log 批量写入。不要默认手动串联分步脚本；分步脚本只用于调试。
4. 如果返回 `route=ask_user`，直接原样发送 `confirmation_question` 字段内容作为回复（该字段已内含 `INFO_ALCHEMIST=TRUE` 激活标记，**不要额外再加一次**），不联网搜索、不输出报告。若宿主无法保存 `confirmation_round`，至少继续传 `--previous-query` 和 `--reply`。
5. 如果用户意图已经明确，宿主模型可以先生成 `ai_search_plan` 草案，再把它和原始问题一起传给正式入口；脚本会校验、去泛词、补证据轴和回填模板 query。不要把 AI 草案直接交给搜索脚本。开放候选池问题由脚本规则推断为 `search_strategy=candidate_discovery`，先发现候选产品/工具/方向再对比；明确目标、最新动态、SEO/转化/选择/复盘问题保持 `normal_evidence`，不要人为加第二轮搜索。
6. 如果返回 `route=formal_report_context`，基于返回的 `search_plan`、`search_result`、`synthesis` 写完整报告；完整原始结果也会写入 `run_log_path`。
7. 不使用 Brave / `web_search` / `web_fetch` / 手工打开网页作为替代搜索。若误用了其他搜索工具，这些结果必须作废，并用正式入口重新跑。
8. 联网搜索全部失败时，立即输出中文失败诊断，并明确写“本轮联网搜索全部失败，不能生成证据报告”。
9. 最终回复第一行必须是 `INFO_ALCHEMIST=TRUE`。用户可见层不要展示底层搜索提供方名称、API key 名称、脚本名或英文 snake_case 字段名。
10. 中文信息炼金报告必须完整保留这些标题：`# 信息炼金报告`、`## 核心判断`、`## 决策问题`、`## 专家判断`、`## 候选行动`、`## 高价值证据`、`## 缺失的证据`、`## 下一步行动`。顶层标题不按问题类型改名，但必须按 `formal_run.py` 返回的 `report_mode_guide` 调整每节内部写法。`## 决策问题` 第一行写 `报告模式：<模式名>`，第二行写 `决策对象：<topic>`；报告模式只用于内部结构选择，HTML 首屏不展示模式标签。文字版每个 `##` 模块之间必须用独立一行 `---` 分割，并在分割线上下各保留一个空行。发送给用户时，标题行必须替换成 `record_final_output.py` 返回的 linked_heading。飞书不稳定支持“Markdown 标题内嵌链接”，所以 linked_heading 是标题块，例如：第一行 `# 信息炼金报告`，下一行 `[点击查看->可视化《信息炼金报告》](http://127.0.0.1:<自动选择端口>/<run_id>.html)`。不要直接发 `file://` 到飞书，飞书会显示蓝色但拦截打开。
11. 写好最终 Markdown 报告后，发送前必须运行：`python3 $SKILL_DIR/scripts/record_final_output.py --run-id "<run_id>" --file "<final_report.md>"`。脚本会把 Markdown 文字版归档到 `info-alchemist/reports/信息炼金报告-<决策对象>-YYYYMMDD.md`，生成短文件名 HTML：`info-alchemist/reports/html/<run_id>.html`，并在 stdout 返回带 `linked_heading` 的完整 `user_visible_text`。宿主直接发送完整 `user_visible_text`，不要再改写成短版。
12. `## 专家判断` 必须放在 `## 候选行动` 上方，优先使用 `synthesis.expert_judgment` 和 `search_intent=expert_signal` 的结果，展示领域专家/高影响力实践者是谁、他们关注的问题、以及他们现在采用的解法。推荐表头：`| 领域专家 | 专家/实践者信号 | 来自渠道 | 为什么选 | 他们关注的问题 | 当前解法 | 对我们的启发 | 可信度 |`。`领域专家` 必须写出可称呼的专家主体：优先写可验证的人名；没有明确人名时，写可识别的机构、社区或实践者群体，例如“AI 工具站独立开发者”“Reddit 的目标用户群体”“小红书 AI 副业博主”；只有连群体主体也无法识别时，才写“未识别到明确专家主体”。可信度应按主体具体程度、来源强度和证据相关性调整；`为什么选` 必须说明该专家主体与本领域、问题、解法或实践经验的关系。
13. `## 高价值证据` 必须用表格展示，表头固定为 `| 行动影响 | 证据方向 | 发现了什么 | 怎么改变决策 | 来源 | 证据质量 | VOI |`，来源列必须有可点击 Markdown 链接；后 2 列是单条证据评分，统一写 0-100 整数，推荐格式 `N/100`。评分标准见 `$SKILL_DIR/references/evidence_scoring.md`：证据质量看来源可信和可追溯性，VOI 看是否会改变默认行动或优先级。
14. 判断页只显示结论强度，不显示整份报告总分；`## 缺失的证据` 只写缺什么和为什么限制判断，不展示证据覆盖度分数表；`## 下一步行动` 直接写具体优先行动，不展示 VOI × 可验证性矩阵。
15. 如果 `synthesis.social_platform_signals` 非空，`## 高价值证据` 内必须新增 `### 社交平台` 小节，用表格按小红书、X、Reddit 展示平台、主要看什么、这轮发现、对行动的影响、可信度和来源；推荐表头：`| 社交平台 | 主要看什么 | 这轮发现 | 对行动的影响 | 可信度 | 来源 |`；不要命名为“四渠道信号矩阵”。

闸门结果：
- 完整 VOI 流程：机会判断、值不值得做、要不要做、SEO/竞品/市场信号、信息堆积、行动复盘。
- 行动目的确认：最新新闻、趋势、动态、参考对象、看看有什么机会等开放集合查询；用户确认搜索结果要改变哪类行动前不联网搜索。研究维度明确不等于决策明确。
- 轻量放行：仅限天气、汇率、单个官方页面、明确事实核验等普通事实查询。新闻/趋势/动态/竞品/市场/产品方向查询不属于轻量放行，确认行动目的后必须使用完整报告骨架。

不触发闸门：翻译、润色、改写、明确执行任务且不需要联网搜索。
```
